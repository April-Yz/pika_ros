# 新旧数据结构与转换方案说明

## 1. 目的

本文档用于说明当前 `pipline.sh` 新采集链路下的数据结构，与原始 `data_tools` 默认数据结构之间的差异。

本文档只描述数据结构和后续转换方案，不修改任何现有源码。

## 2. 原始默认结构

原始 `multi_pika_teleop` 转换链默认按下面的数据结构理解数据：

### 相机

- `camera/color/pikaGripperDepthCamera_l`
- `camera/color/pikaGripperDepthCamera_r`
- `camera/color/pikaGripperFisheyeCamera_l`
- `camera/color/pikaGripperFisheyeCamera_r`
- `camera/depth/pikaGripperDepthCamera_l`
- `camera/depth/pikaGripperDepthCamera_r`
- `camera/pointCloud/...` 可选

### 机械臂

- `arm/jointState/masterLeft`
- `arm/jointState/masterRight`
- `arm/jointState/puppetLeft`
- `arm/jointState/puppetRight`
- `arm/endPose/masterLeft`
- `arm/endPose/masterRight`
- `arm/endPose/puppetLeft`
- `arm/endPose/puppetRight`

### 定位与夹爪

- `localization/pose/pika_l`
- `localization/pose/pika_r`
- `gripper/encoder/pikaSensor_l`
- `gripper/encoder/pikaSensor_r`
- `gripper/encoder/pikaGripper_l`
- `gripper/encoder/pikaGripper_r`

原始转换脚本默认要求 fisheye 字段存在，因此默认配置不能直接无缝适配你现在的新采集结构。

## 3. 当前新采集结构

你现在的采集链路来自 `pipline.sh` 中的 buffered 10Hz no-fisheye 方案，数据实际落盘结构已经变成：

### 相机

- `camera/color/pikaGripperDepthCamera_l`
- `camera/color/pikaGripperDepthCamera_r`
- `camera/color/myD435`
- `camera/depth/pikaGripperDepthCamera_l`
- `camera/depth/pikaGripperDepthCamera_r`

### 不再采集的相机项

- 不采集 `camera/color/pikaGripperFisheyeCamera_l`
- 不采集 `camera/color/pikaGripperFisheyeCamera_r`
- 不采集 D435 depth
- 不采集 D435 point cloud

### 机械臂、定位、夹爪

这部分与原始双臂遥操采集结构基本一致，仍保留：

- `arm/jointState/*`
- `arm/endPose/*`
- `localization/pose/*`
- `gripper/encoder/*`

## 4. 新旧结构的核心区别

### 4.1 相机组成不同

原始默认结构：

- 双 gripper RGB
- 双 fisheye RGB
- 双 gripper depth

当前新结构：

- 双 gripper RGB
- 单额外头部 RGB 相机
- 双 gripper depth

也就是说，新的相机集合本质上是：

- 删除 fisheye
- 新增头部 RGB 相机

### 4.2 目录名与语义名不同

当前实际采集目录名是：

- `camera/color/myD435`

但从任务语义上，这一路数据更适合被视为：

- `headCam`

因此后续新的转换方案中，建议采用下面的映射关系：

- 原始落盘目录：`camera/color/myD435`
- 转换后的语义字段：`camera/color/headCam`

这个映射只发生在新转换链路中，不要求你修改现有采集目录名。

同时，当前三路 RGB 的语义必须固定写清楚：

- `camera/color/pikaGripperDepthCamera_l` 是左手腕 RGB
- `camera/color/pikaGripperDepthCamera_r` 是右手腕 RGB
- `camera/color/myD435` 是头部 RGB

后续处理脚本中，这三路图像应分别映射为：

- 左手腕 RGB
- 右手腕 RGB
- `headCam`

### 4.3 原始默认 type 不再适配

原始 `multi_pika_teleop` 的配置默认包含：

- `pikaGripperFisheyeCamera_l`
- `pikaGripperFisheyeCamera_r`

而你当前的数据目录中没有这两个目录，因此直接使用原始 `multi_pika_teleop` 去做同步和 HDF5 转换，会出现以下问题之一：

- 找不到 fisheye 目录
- 同步阶段判定数据不完整
- HDF5 转换字段与实际目录不匹配

## 5. 建议采用的新转换视图

建议把当前数据解释成一个新的逻辑数据类型：

- 名称建议：`multi_pika_teleop_headcam_no_fisheye`

这个新逻辑类型的相机字段建议定义为：

### 颜色图像

- `camera/color/leftWrist` 对应原始目录 `camera/color/pikaGripperDepthCamera_l`
- `camera/color/rightWrist` 对应原始目录 `camera/color/pikaGripperDepthCamera_r`
- `camera/color/headCam` 对应原始目录 `camera/color/myD435`

### 深度图像

- `camera/depth/leftWrist` 对应原始目录 `camera/depth/pikaGripperDepthCamera_l`
- `camera/depth/rightWrist` 对应原始目录 `camera/depth/pikaGripperDepthCamera_r`

如果你希望保持与原仓库字段名尽量一致，也可以只做最小映射：

- `camera/color/pikaGripperDepthCamera_l`
- `camera/color/pikaGripperDepthCamera_r`
- `camera/color/headCam`
- `camera/depth/pikaGripperDepthCamera_l`
- `camera/depth/pikaGripperDepthCamera_r`

当前更推荐这个最小映射版本，因为：

- 对已有 gripper 相机命名侵入最小
- 只把 `myD435` 替换成更语义化的 `headCam`
- 后续脚本更容易和现有目录结构对应

## 6. 推荐的最终映射

建议后续新转换脚本采用以下映射：

| 实际采集目录 | 转换后 key |
| --- | --- |
| `camera/color/pikaGripperDepthCamera_l` | `camera/color/pikaGripperDepthCamera_l` |
| `camera/color/pikaGripperDepthCamera_r` | `camera/color/pikaGripperDepthCamera_r` |
| `camera/color/myD435` | `camera/color/headCam` |
| `camera/depth/pikaGripperDepthCamera_l` | `camera/depth/pikaGripperDepthCamera_l` |
| `camera/depth/pikaGripperDepthCamera_r` | `camera/depth/pikaGripperDepthCamera_r` |

这样做之后：

- 原始采集目录不变
- 新 HDF5 中不再出现 `myD435`
- 下游训练和读取统一面向 `headCam`

如果后续要进一步处理成 RobotWin 风格数据，推荐再做一次图像别名映射：

| 语义相机 | RobotWin 输出字段 |
| --- | --- |
| 头部 RGB | `cam_high` |
| 左手腕 RGB | `cam_left_wrist` |
| 右手腕 RGB | `cam_right_wrist` |

## 7. 新转换方案的原则

为了满足“不改源代码”的要求，后续转换建议遵守以下原则：

### 7.1 不修改现有采集脚本

不改以下内容：

- `pipline.sh`
- `start_s5_buffered_10hz_no_fisheye_capture.bash`
- `start_s6_buffered_10hz_no_fisheye_capture.bash`
- 现有 `launch` 文件

### 7.2 不修改原始 `src/data_tools` 源码

不直接改：

- `src/data_tools/scripts/data_sync.py`
- `src/data_tools/scripts/data_to_hdf5.py`
- `src/data_tools/config/*`

### 7.3 单独新增一套转换入口

建议新增独立文件，例如：

- `scripts/custom_data_tools/multi_pika_teleop_headcam_no_fisheye_data_params.yaml`
- `scripts/custom_data_tools/data_sync_headcam_no_fisheye.py`
- `scripts/custom_data_tools/data_to_hdf5_headcam_no_fisheye.py`

这套脚本专门负责：

- 读取当前真实目录结构
- 把 `myD435` 映射成 `headCam`
- 跳过 fisheye
- 生成新的 `sync.txt`
- 生成新的 `data.hdf5`

## 9. 当前结论

当前可以明确确定：

- 你的新采集结构已经不再适合直接使用原始 `multi_pika_teleop` 配置
- 最合理的做法是新增一条独立转换链
- 新转换链中应把 `myD435` 统一映射为 `headCam`
- fisheye 应从新转换配置中完全移除

这部分结论已经稳定，可以据此继续写新的转换脚本，不需要反复改方向。
