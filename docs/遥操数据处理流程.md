# 遥操数据处理流程

本文档记录当前 `pika_ros` 遥操采集数据到 RobotWin 训练数据的处理流程。

示例任务：

- task name: `pnp_bread`
- instruction: `Pick up two breads, then place them onto the blue plate.`
- 原始数据目录：`/home/piper/agilex/pnp_bread`

## 0. 环境

建议使用系统 Python 跑和 OpenCV 相关的脚本：

```bash
/usr/bin/python3
```

如果使用 conda，环境至少需要：

```bash
conda create -n robotwin-data python=3.10 -y
conda activate robotwin-data
conda install -c conda-forge numpy h5py opencv scipy pyyaml -y
```

- `/home/piper/agilex/pnp_bread/good/episodeX`
- `/home/piper/agilex/pnp_bread/bad/episodeX`

筛选规则：

- 缺少三路 RGB 视角则进入 `bad`

```bash
/usr/bin/python3 ~/pika_ros/scripts/curate_teleop_episodes.py manual-review \
  --task-name pnp_tray

```

播放窗口快捷键：

- `g`：标记为好，保留在 `good`
- `m`：标记为中等质量，移动到 `medium`
- `n`：下一个，等价于保留
- `b`：标记为坏，移动到 `bad`
- `p`：上一个
- `r`：重播当前视频
- `a`：降低播放速度
- `d`：提高播放速度
- `space`：暂停/继续
- `q`：退出人工筛选

窗口会实时显示：

- 当前是第几个 `good` 视频
- 本轮人工审核总共有多少个 `good` 视频
- 已经手动标注了多少个
- 其中有多少个被判为 `good / medium / bad`
- 当前播放速度

说明：

- `medium` 默认和 `bad` 一样，不参与后续 HDF5 处理。
- 这样可以把“明显坏数据”和“边缘质量数据”分开保留。

日志位置：


如果你想单独检查某个 `episode` 为什么有帧被删除，可以直接看“被删帧”和“上一保留帧 / 上一原始帧”的 7 维差值：

```bash
/usr/bin/python3 ~/pika_ros/scripts/inspect_trim_static_episode.py \
  /home/piper/agilex/pnp_bread/good/episode147 \
  --motion-threshold 0.0001 \
  --gripper-source sensor \
  --limit 30 \
  --output-json /home/piper/agilex/pnp_bread/debug_trim/episode147_inspection.json

/usr/bin/python3 ~/pika_ros/scripts/inspect_trim_static_episode.py \
  /home/piper/agilex/place_bread_basket/good/episode10 \
  --motion-threshold 0.0001 \
  --limit 30 \
  --output-json /home/piper/agilex/place_bread_basket/debug_trim_episode10_inspection.json
```

这个检查脚本会输出：

- 每个被删帧相对“上一保留帧”的左右手 7 维绝对变化量之和
- 每个被删帧相对“上一原始帧”的左右手 7 维绝对变化量之和
- 每一维的详细差值：
  - `x, y, z, roll, pitch, yaw, gripper_distance`

这样你可以区分：

- 机器人确实几乎没动
- 机器人在连续微动，但累计到上一保留帧的变化仍然低于阈值
- 某个维度的单位或权重让阈值判断不符合直觉

规则：

- 基于头部相机时间线对齐数据
- `--gripper-source robot` 时：
  - 每只手使用 7 维：
    - `x, y, z, roll, pitch, yaw, robot_gripper_distance`
- `--gripper-source sensor` 时：
  - 每只手使用 8 维：
    - `x, y, z, roll, pitch, yaw, robot_gripper_distance, sensor_gripper_distance`
  - 也就是不会用 sensor 替换 robot gripper，而是二者都参与“是否静止”的判断
- 左右手分别计算对应维度绝对变化量之和
- 如果某一帧相对上一个保留帧的左右手最大“对应维度绝对变化量之和”小于阈值，则删除
- 第一帧和最后一帧始终保留

关于旋转量：

- `roll, pitch, yaw` 直接读取原始 `endPose` JSON 中的值
- 单位是 `rad`
- 当前脚本不会把旋转量换算成米，也不会做额外归一化
- 也就是说，当前位置变化 `m`、旋转变化 `rad`、robot gripper 变化量、sensor gripper 变化量会直接一起参与上面的求和阈值
- 因此 `--motion-threshold` 目前是一个工程上的混合阈值，不是严格物理同量纲的距离
- 所以 `--motion-threshold 0.0001` 不能直接理解成“严格的 0.1 mm 阈值”
- 只有在旋转项和夹爪项几乎不变时，它才大致接近“位置总变化量约 0.1 mm”
- 如果后面你觉得旋转量权重过大或过小，再单独给旋转项加系数会更稳

  如果只有位置变化：

  - 阈值是 0.0001
  - 位置单位是 m
  - 所以单独一维大约要到 0.0001 m
  - 也就是 0.1 mm

  如果只有朝向变化：

  - roll/pitch/yaw 单位是 rad
  - 单独一维要到 0.0001 rad
  - 换算成角度：
      - 0.0001 * 180 / pi ≈ 0.00573°
  - 也就是大约 0.0057 度
为什么看起来“视频不卡”，但仍然会删掉较多帧：

- 这个规则不是按视觉卡顿判断，而是按左右手状态量变化是否足够大判断
- 如果机器人在连续微动，但相邻若干帧的累计状态变化始终低于阈值，这些帧仍然会被删
- 尤其在 10 Hz 采集下，很多慢速接近、微调、夹持保持阶段，本来就会出现较高比例的“近静止帧”
- 如果你想确认是否删得过头，建议先用 `inspect_trim_static_episode.py` 检查一个具体 episode 的被删帧明细

裁剪输出目录中的 `camera_overview.mp4`：

- 现在不会再直接复制原始 episode 里的 `camera_overview.mp4`
- 脚本会根据裁剪后保留下来的图片重新生成新的 `camera_overview.mp4`
- 所以 `good_trimmed_*` 目录里的总览视频已经对应“删帧之后”的结果

输出报告：

```bash
/home/piper/agilex/pnp_bread/good_trimmed_01mm/static_trim_report.json
```

报告中包含每个 episode：

- 原始帧数
- 剩余帧数
- 删除帧数
- 剩余百分比
- 输出目录
